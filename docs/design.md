## Problem Statement

The existing web-ai-test project uses manually maintained model config files (model_onnx.js, model_tflite.js) that are painful to keep up-to-date as models change daily. There is no open-source, community-driven, multi-runtime browser ML benchmark with HuggingFace integration, shareable results, or user roles. The space is wide open for a tool that serves browser/runtime engineers at Intel, Qualcomm, Google, Microsoft and web developers exploring client-side ML.

## What Makes This Cool

- **HuggingFace as a live model registry.** Models auto-discovered from HF orgs/collections. Zero manual config forever.
- **Recipes.** User-curated model+backend+datatype selections stored in DB. Shareable via URL. Anyone can click and run.
- **Hidden leaderboard.** Community submits benchmark results. Only Intel/Admin sees the aggregated hardware performance rankings. The benchmark is the Trojan horse; the data is the product.
- **Multi-runtime, multi-backend.** ONNX Runtime Web + LiteRT.js across Wasm, WebGPU, WebNN (CPU/GPU/NPU). Nobody else covers this matrix.
- **Five-tier role system.** Anonymous, Member, Partner, Intel, Admin with progressive feature access.

## Constraints

- Client-side only inference (no cloud AI)
- SvelteKit 5 + TypeScript + Vite
- Supabase for auth and database
- Deployed on Vercel
- HTTPS required (SharedArrayBuffer for multi-threaded Wasm)
- COOP/COEP headers for cross-origin isolation
- Must work with ONNX Runtime Web and LiteRT.js runtimes
- Model metadata parsed from file binaries (onnx-proto, flatbuffers) or runtime session objects

## Premises

1. Model input metadata (shape, dtype) is parsed directly from the model file binary (onnx-proto for ONNX, flatbuffers for TFLite) or extracted via the runtime after model load. HuggingFace API provides model listing + file paths; input specs come from the model files themselves. System caches parsed metadata in DB per model.
2. Supabase for auth (5 roles), recipes, result collection.
3. Auth + recipes + HF auto-discovery built together in v1 (Full Pipeline approach).
4. Hidden leaderboard (Intel/Admin-only) justifies collecting results from all tiers. Users see "save results" — never "leaderboard."
5. Existing execution logic (ort_utils, litert_utils, nn_utils) is leveraged but verified for bugs/design issues before porting. Not a blind copy.

## Approaches Considered

### Approach A: Recipes-First MVP
Auth + recipes + manual model curation. 3-5 hardcoded models per runtime. Use runtime session metadata (no binary parsing). Add HF discovery later.
- Effort: M (human: ~2 weeks / CC: ~2-3 hours)
- Risk: Low
- Pros: Fastest to validate. Lowest risk.
- Cons: Limited catalog. Manual curation burden.

### Approach B: Full Pipeline (CHOSEN)
Auth + recipes + HF auto-discovery + binary metadata parsing from day one. Both ONNX and TFLite supported. Metadata cached in Supabase.
- Effort: L (human: ~4-5 weeks / CC: ~5-6 hours)
- Risk: Medium (HF API edge cases, metadata parsing for unusual models)
- Pros: Complete vision from launch. Scales to hundreds of models. No "phase 2" needed.
- Cons: More surface area to debug. HF API rate limits need handling.

## Recommended Approach

**Approach B: Full Pipeline.** Build auth, recipes, and HuggingFace auto-discovery together from day one. The metadata parsing layer (onnx-proto + flatbuffers) caches model input specs in Supabase so the recipe/model browser works without downloading full model files. Models with unparseable metadata are flagged as "needs manual config" and gracefully handled.

## HuggingFace Integration

### Predefined Sources
- https://huggingface.co/Xenova/models
- https://huggingface.co/onnx-community/models
- https://huggingface.co/webnn/models
- https://huggingface.co/webgpu/models
- https://huggingface.co/litert-community/models

### Model Discovery Flow
1. Query HF API for models in configured orgs/collections
2. List model files (*.onnx, *.tflite) with sizes
3. Parse model file binary for input tensor metadata (shape, dtype)
4. Cache parsed metadata in Supabase (model_id, file_path, inputs, data_type, size)
5. Models with unparseable metadata flagged as "unverified"

### Metadata Parsing
- ONNX: `onnx-proto` (protobufjs) — decode ModelProto, traverse graph.input[] for shape/dtype
- TFLite: `flatbuffers` + compiled TFLite schema — read subgraphs[0].tensors[i] for shape/type
- Fallback: runtime session metadata after model load (requires full download)

## Recipe System

A "recipe" is a named, shareable collection of model + backend + datatype selections:
- Stored in Supabase with owner, visibility (personal/public), and model list
- Each recipe entry: HF model ID, file path, detected datatype, selected backends
- Shareable via URL: `webai.run/r/{recipe-slug}`
- Any user (including Anonymous) can run a public recipe
- Members can create, edit, publish recipes
- Recipes can be run for 1, 10, 20, 50, 100, 500, 1000, or 10000 inference iterations

## Model Download Strategy

Download sources tried in order (same pattern as web-ai-test):
1. HuggingFace CDN (primary)
2. HuggingFace Mirror (hf-mirror.com) — for users behind firewalls
3. Local `/static/models/` fallback (for development/offline)

Client-side caching:
- **OPFS (Origin Private File System)** — primary cache for large model files
- **Cache API** — fallback when OPFS unavailable
- Progressive download with ReadableStream for progress tracking
- Models cached persistently across sessions (no re-download)

## Custom Model Upload

Users can upload and test their own models without adding them to HuggingFace:
- `/custom` route — upload .onnx, .tflite, or .litertlm files. Runtime auto-detected from file extension. Auto-detect input specs, run benchmark.
- Custom models are NOT added to the shared catalog (personal use only)
- Input metadata extracted from the uploaded file binary (same onnx-proto/flatbuffers pipeline)

## Runtime Version Management

- ONNX Runtime Web and LiteRT.js loaded dynamically (not bundled)
- Default: latest dev version
- Version selector: show last 50 versions grouped by "dev" and "stable"
- Sources: jsDelivr CDN for version listing and loading
- Guard against double-loading (window.__ortLoaded__, window.__litertLoaded__ flags)
- WebNN: Origin Trial tokens in app.html for access in supported browsers

## Metadata Parsing Strategy (RESOLVED)

Parsing happens **client-side at recipe creation time and on first model browse:**
1. When a user adds a model to a recipe (or browses a model for the first time), the client fetches the model file
2. For ONNX: parse full protobuf with onnx-proto to extract graph.input[] (shape, dtype)
3. For TFLite: parse flatbuffer to extract subgraphs[0].tensors (shape, type)
4. Parsed metadata is POSTed to Supabase and cached (subsequent users skip parsing)
5. For very large models where full download is impractical: fall back to runtime session metadata (load model, read inputNames/inputMetadata, then proceed to benchmark)
6. Models with unparseable metadata: appear in browser with a warning badge. Can be added to recipes but require user to manually specify input shape/dtype before running.

## HF API Rate Limit Strategy

- **Background refresh:** Supabase Edge Function runs daily to sync model listings from configured HF orgs
- **Pagination:** HF API paginated (100 models/page), fetched sequentially with 1s delay between pages
- **Cache TTL:** Model list cached for 24 hours. Metadata cache has no expiry (model binaries don't change)
- **User-triggered refresh:** Members can trigger a re-sync of their custom HF orgs (rate limited to 1/hour)
- **Fallback:** If HF API is down, serve from cached data. Show "last synced: {timestamp}" in UI

## Role Assignment Flow

- **Signup → Member** (default for all new signups)
- **Member → Partner:** Admin manually promotes via admin panel. Verification: work email domain (@google.com, @microsoft.com, @huggingface.co, approved ISV domains) OR admin vouches
- **Member → Intel:** Admin promotes. Verification: @intel.com email domain
- **Intel → Admin:** Existing admin grants via Supabase custom claims
- **Implementation:** Supabase custom claims in JWT: `{ role: "anonymous" | "member" | "partner" | "intel" | "admin" }`
- **RLS policies:** Each table has row-level security policies checking `auth.jwt() -> 'role'`

## Recipe Behavior

- Recipes define exactly which model+backend combinations to benchmark
- Users running a shared recipe execute those exact combinations (no override from the recipe URL)
- Users can fork a recipe to create their own modified version
- Backend selection in recipe: user picks from available backends at creation time
